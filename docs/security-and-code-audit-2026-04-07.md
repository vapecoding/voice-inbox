## Основные findings

### 1. High — глобальный поиск по всем транскриптам без tenant-scoping
- Поиск в боте вызывается из текстовых сообщений в [../bot.py](../bot.py).
- SQL-поиск не фильтрует результаты по `telegram_chat_id` в [../db.py](../db.py).
- Риск: любой пользователь бота может искать по чужим записям, если бот используется более чем одним человеком или чатом.

Подтверждение:
- [../bot.py](../bot.py) — `handle_search()`
- [../db.py](../db.py) — `search_recordings()`

### 2. High — веб-панель работает по HTTP и ставит cookie без `Secure`
- Waitress слушает `0.0.0.0` в [../main.py](../main.py).
- Fallback URL собирается как `http://...` в [../config.py](../config.py).
- Сессионная cookie выставляется без `secure=True` в [../web.py](../web.py).
- Риск: пароль и `session_token` могут быть перехвачены на сети.

Подтверждение:
- [../main.py](../main.py)
- [../config.py](../config.py)
- [../web.py](../web.py)
- [DEPLOYMENT.md](./DEPLOYMENT.md)

### 3. High — у бота нет allowlist/ACL на вход
- Все приватные сообщения принимаются без проверки `user_id` или `chat_id`.
- Ограничение доступа к боту по user id отдельно перечислено как отсутствующее в [PROJECT_OVERVIEW.md](./PROJECT_OVERVIEW.md).
- Риск: любой нашедший бота может расходовать Deepgram/Groq и наполнять локальную БД.

Подтверждение:
- [../bot.py](../bot.py)
- [PROJECT_OVERVIEW.md](./PROJECT_OVERVIEW.md)

### 4. Medium — веб-логин без rate limit и защиты от brute force
- Проверка логина сводится к сравнению одного пароля.
- Нет задержек, lockout, счётчика попыток или IP throttling.
- Риск: перебор пароля для публичной панели.

Подтверждение:
- [../web.py](../web.py)

### 5. Medium — сессии веба не имеют server-side expiration
- Токены хранятся в in-memory `set`.
- `max_age` ограничивает только клиентскую cookie, но не серверную валидность токена.
- Риск: украденный токен валиден до logout или рестарта процесса.

Подтверждение:
- [../web.py](../web.py)

### 6. Medium — схема `summaries` не защищает инварианты данных
- В таблице `summaries` нет `CHECK`, который требовал бы, чтобы был задан ровно один из `recording_id` или `topic_id`.
- Нет ограничения, препятствующего дублированию summary для одной и той же сущности.
- Риск: логически битые строки, дубли, неоднозначное поведение при чтении.

Подтверждение:
- [../db.py](../db.py)

### 7. Medium — генерация summary допускает гонки и двойные записи
- Логика сначала читает существующее summary, потом делает внешний запрос, потом пишет результат.
- При двух быстрых callback-ах возможны двойной запрос к Groq и дубли в БД.
- Риск: лишние расходы и нарушение целостности данных.

Подтверждение:
- [../bot.py](../bot.py)
- [../db.py](../db.py)

### 8. Medium — веб-сервер запускается daemon-thread’ом без fail-fast
- Веб поднимается в `daemon=True` потоке.
- Если Waitress не стартует или падает, бот может остаться жив, а веб-панель исчезнет.
- Риск: частично сломанное состояние процесса без явного падения сервиса.

Подтверждение:
- [../main.py](../main.py)
- [PROJECT_OVERVIEW.md](./PROJECT_OVERVIEW.md)

### 9. Low — хрупкий startup-конфиг
- `WEB_PORT` парсится через голый `int(...)` без аккуратной валидации.
- `DATABASE_PATH` принимается как есть; относительный путь к БД может привести к созданию другой SQLite-базы при смене `cwd`.
- Риск: нестабильный старт, сложная диагностика, ложные инциденты по данным.

Подтверждение:
- [../config.py](../config.py)
- [../db.py](../db.py)

### 10. Low — уязвимая версия Flask в зависимостях
- В `requirements.txt` зафиксирован `Flask==3.1.0`.
- `pip-audit -r requirements.txt` на 2026-04-07 показал:
  - `CVE-2025-47278`, исправление: `Flask 3.1.1`
  - `CVE-2026-27205`, исправление: `Flask 3.1.3`
- Важная оговорка: текущее приложение не использует Flask session API, поэтому это подтверждённый dependency issue, но не обязательно exploitable bug именно в этой кодовой базе.

Внешние источники:
- https://nvd.nist.gov/vuln/detail/CVE-2025-47278
- https://nvd.nist.gov/vuln/detail/CVE-2026-27205

## Рекомендуемый порядок исправлений
1. Закрыть доступ к боту allowlist’ом и сделать search chat-scoped.
2. Убрать публичный HTTP: либо reverse proxy + HTTPS, либо bind на localhost.
3. Усилить веб-логин: rate limit, задержки, защита cookie, серверная экспирация.
4. Добавить ограничения целостности и миграции для SQLite.
5. Обновить `Flask` минимум до `3.1.3`.

## Следующий шаг
