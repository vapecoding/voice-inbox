# AGENTS.md

## Что это за проект

`Voice Inbox` — Telegram-бот для голосовых заметок с веб-панелью.

Бот:
- принимает `voice` в Telegram,
- отправляет аудио в Deepgram,
- сохраняет транскрипцию в SQLite,
- по кнопке генерирует summary через Groq,
- даёт поиск по тексту,
- поддерживает режим тем (`/topic`).

Веб:
- показывает ленту записей и тем,
- даёт просмотр записи/темы,
- защищён одним паролем через cookie-сессию,
- позволяет удалять записи и темы.

## Ключевые файлы

- `main.py` — запуск бота и веба
- `config.py` — env-конфиг
- `db.py` — SQLite и CRUD
- `bot.py` — Telegram-логика
- `web.py` — Flask-приложение
- `templates/` — HTML-шаблоны
- `PLAN_V3.md` — основной план реализации и практические нюансы

## Что важно знать будущему агенту

- Локальный входной файл для автоматизации — `.env` в корне проекта.
- На Windows `python` может быть alias из `WindowsApps`; часто безопаснее использовать `py`.
- На Windows инструмент `apply_patch` может падать с `Io(Os { code: 206, kind: InvalidFilename, message: "The filename or extension is too long." })`, если агент пытается применить слишком большой patch за один раз.
- При первой настройке сервера root-пароль может оказаться истёкшим и потребовать forced password change.
- OpenSSH for Windows чувствителен к ACL на `~/.ssh/config` и приватных ключах.
- При генерации SSH-ключей на Windows важно не создать случайно passphrase из-за неправильного quoting.
- Для деплоя в серверный `.env` полезно добавлять `PUBLIC_BASE_URL`.

## Правила редактирования на Windows

- Не копи большой diff до конца сессии. Применяй изменения маленькими шагами сразу после того, как понял локальный кусок работы.
- Не пытайся одним `apply_patch` целиком заменить большой HTML/CSS/JS-шаблон, длинный markdown-документ или несколько крупных файлов сразу.
- Для больших правок разбивай работу минимум так: сначала базовый каркас, потом отдельные секции или страницы, потом точечные добивки.
- Если меняются шаблоны в `templates/` или крупные документы в `docs/`, предпочитай несколько маленьких patch-вызовов вместо одного большого.
- После каждого заметного patch-шага перечитай изменённый файл и только потом переходи к следующему блоку.
- Если `apply_patch` вернул `code: 206` или `InvalidFilename`, не повторяй ту же большую вставку. Сразу остановись и разбей изменение на более мелкие куски.
- Если правка явно большая, заранее предупреди пользователя, что будешь вносить её поэтапно, чтобы не упереться в лимит Windows.

## Отчёты

Отчёты должны быть не только итоговыми, но и процессными:
- что делалось,
- что сломалось,
- как это чинилось,
- что проверено,
- какие артефакты изменились.

Секреты, токены, пароли и приватные ключи в отчёты не писать.
